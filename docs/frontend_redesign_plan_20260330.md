# 前端重设计冻结计划（2026-03-30）

## 参考项目
本重设计方案参考项目：[biliLive-tools](https://github.com/renmu123/biliLive-tools)  
该项目提供了直播录制工具的优秀交互设计与信息架构参考，本方案将借鉴其：
- 直播录制页的列表交互与批量操作模式
- 看板页的指标展示与快捷操作布局
- 整体视觉层次与状态反馈机制

**注意**：设置页不强制还原参考项目结构，将以本项目配置模型为准进行适配。

## 一、目标
在不破坏现有后端接口与已通过验收脚本的前提下，完成 Web 前端三大标签页重设计：

1. 直播录制页：高还原参考项目交互与信息密度。
2. 看板页（Phase 3）：高还原参考项目的交互层级与视觉表达。
3. 设置页（Phase 4）：以本项目配置模型为唯一准绳，适配优先，不强制还原参考项目设置结构。

## 二、总体策略

1. 技术路线：Vue3 + Naive UI + Vite 前端子工程。
2. 对接方式：通过 API 适配层连接现有 FastAPI 接口。
3. 数据刷新：保持 3 秒轮询（仅在标签激活时执行），本期不引入 SSE。
4. 风险控制：后端最小改动，保持 scripts/phase5_regression.sh、scripts/phase6_config_validation.sh、scripts/phase6_network_validation.sh 可持续通过。

## 三、阶段规划

### Phase 0 - 契约冻结与差异映射（起点）
1. 冻结当前接口与字段：/api/v1/tasks、/api/v1/dashboard、/api/v1/config/settings、/api/v1/config/update-settings、/api/v1/summary。
2. 输出“参考交互 -> 本项目接口”映射表，区分可复用项、降级项、补强项。
3. 保持状态语义不变：offline|monitoring|live_not_recording|recording|stopping|failed 及其 live_status/recording_status 映射。

### Phase 1 - 前端骨架与设计系统（依赖 Phase 0）
1. 建立三主标签页信息架构：直播录制、看板、设置。
2. 建立统一 token：颜色、间距、阴影、圆角、状态色、动效。
3. 建立响应式规范：桌面（>=1080）与移动（<1080）双断点。

### Phase 2 - 直播录制页高还原实现（依赖 Phase 1）
1. 实现筛选、排序、分页、列表视图、列显隐与批量操作。
2. 实现新增/编辑/批量操作弹窗与统一反馈机制（toast + 字段级错误）。
3. 持久化页面偏好：分页大小、排序字段、列配置。
4. 本期不实现卡片视图及视图切换入口，统一采用列表视图。

### Phase 3 - 看板页高还原实现（依赖 Phase 1，可与 Phase 2 并行）
1. 高还原参考项目看板结构：头部刷新、指标卡矩阵、快捷操作区、结果反馈弹窗。
2. 高还原视觉表达：图标渐变、hover 层级、告警态动画、卡片层次。
3. 三列状态看板：正在直播 / 正在录制 / 未开播，支持跳转联动。
4. 数据字段以本项目可用能力为准：任务总数、启用/禁用、录制中、失败、平台分布、运行时长、（按后端可用性接入）。
5. 刷新策略保持 3 秒轮询，离开标签自动暂停。

### Phase 4 - 设置页适配重构（依赖 Phase 1，可与 Phase 2/3 并行）
1. 采用 schema 驱动渲染，复用后端 group/batch/type/choices/min/max/effect 元数据。
2. 分组以本项目配置批次为主：第一批录制核心、第二批网络稳定、第三批通知与凭据预留位。
3. 明确不强制还原参考项目设置结构：不新增后端未支持字段，不引入伪配置项。
4. 强化校验与生效提示：保存前校验、保存后字段级反馈、支持重新读取。
5. 保持数值兼容：int 字段允许 300.0 等整值浮点输入并归一化为整数。

### Phase 5 - 最小后端补强（依赖 Phase 0，可与 Phase 2/3/4 并行）
1. 仅补齐支撑看板高还原展示的必需指标。
2. 建议优先新增：应用运行时长、磁盘空间摘要。
3. 对无法稳定提供的参考专有指标，在 UI 明确降级说明。
4. 保持现有任务与配置接口兼容，不影响既有回归基线。

### Phase 6 - 集成切换与回退策略（依赖 Phase 2/3/4/5）
1. 引入前端版本开关（例如 ?ui=v2 或环境变量）用于灰度切换。
2. 保留旧版页面回退入口，直到新界面验收完成。
3. Docker 构建链加入前端打包产物，继续由 FastAPI 托管静态资源。

### Phase 7 - 验证与验收（依赖 Phase 6）
1. 自动化回归：
   - scripts/phase5_regression.sh
   - scripts/phase6_config_validation.sh
   - scripts/phase6_network_validation.sh
2. 交互验收矩阵：
   - 录制页：筛选、排序、分页、批量、弹窗、列显隐（仅列表视图）。
   - 看板页：高还原结构、视觉、告警态、轮询与跳转联动。
   - 设置页：读取、保存、重载、字段级错误与生效提示。
3. 设备验收：桌面 Chrome + 移动窄屏触控操作可用。

## 五、冻结决策

2. 看板页（Phase 3）：高还原实现。

3. 变更后必须可通过既有 Phase 5/6 回归脚本。

## 九、拍板执行规则（2026-03-31）

拍板结果：1C 2C 3C 4B 5C 6C 7C。

### 9.1 前端版本开关决策矩阵（1C）

1. 生效来源与优先级：
   - 第一优先级：URL 参数 `?ui=`（仅当值在允许集合内时生效）。
   - 第二优先级：浏览器会话记忆（sessionStorage）。
   - 第三优先级：环境变量默认值。
   - 第四优先级：最终兜底 `v1`。
2. 允许集合：由环境变量提供（例如 `v1,v2`）。URL 传入不在集合内时忽略并回退。
3. 回退入口：新界面保留“切回旧版”入口，直到 Phase 7 验收通过后再评估下线。
4. 最小验收用例：
   - 用例 A：环境变量为 `v1`，URL 指定 `?ui=v2` 且 `v2` 在允许集合内，应展示 `v2`。
   - 用例 B：URL 指定非法值，应忽略并按会话记忆/环境变量回退。
   - 用例 C：用户在页面内切换版本后刷新，应保持会话版本。

### 9.2 任务状态到看板三列映射真值表（2C）

1. 基础映射：
   - `recording` -> 正在录制。
   - `live_not_recording` -> 正在直播。
   - `monitoring` -> 未开播。
   - `offline` -> 未开播。
   - `stopping` -> 正在录制（子态标记“停止中”）。
2. `failed` 的归属（按 live/recording 子状态纠偏）：
   - 若 `recording_status` 为 `recording` 或 `stopping`，归入“正在录制”。
   - 否则若 `live_status` 为直播中，归入“正在直播”。
   - 其他情况归入“未开播”。
3. 异常高亮规则：`failed` 始终附加异常标记，不新增第四列。
4. 跳转联动：点击任一列，仅带列对应过滤条件进入录制页，不隐式叠加其他筛选。

### 9.3 3 秒轮询状态机（3C）

1. 调度模型：采用“请求完成后再调度下一次”（completion-driven），基础间隔 3 秒。
2. 首刷时机：标签从非激活切为激活时立即首刷一次。
3. 并发约束：单飞（single-flight），同一标签同一接口同一时刻仅允许 1 个在途请求。
4. 暂停与恢复：离开标签页立即暂停调度；返回标签页立即恢复并触发首刷。
5. 错误退避：连续失败时退避到 3s -> 6s -> 12s -> 24s（上限 30s），成功后恢复 3s。
6. 超时与取消：请求超时进入失败计数；标签失焦时取消在途请求（若实现支持取消）。

### 9.4 设置页 schema 联动与提交语义（4B）

1. 提交策略：仅提交脏字段（dirty patch），未变更字段不提交。
2. 联动策略：由 `effect` 导致的自动赋值字段记为脏字段并纳入提交。
3. 隐藏字段策略：仅当隐藏字段在本次交互中被联动改写时提交；否则不提交。
4. 错误处理：采用字段级错误回传，不做整单回滚；成功字段保持成功状态。
5. 保存后行为：保存成功后触发一次配置重读，前端以服务端返回值回填表单。

### 9.5 录制页批量操作边界（5C）

1. 选择模型：
   - 默认仅“当前页全选”。
   - 用户可二次确认后升级为“当前筛选结果全选”。
2. 筛选变更策略：筛选条件变更后清空“跨页选择”，并给出一次性提示。
3. 执行确认：涉及停录、删除等高风险操作时显示二次确认，确认文案包含影响条数。
4. 部分成功反馈：统一反馈“成功 X 条，失败 Y 条”，并可展开失败明细（任务名 + 原因）。

### 9.6 看板指标口径字典（6C）

1. 指标返回结构：后端返回原始值、单位、采样时间戳、可用性标记；前端统一格式化显示。
2. 运行时长口径：以进程启动时间为起点，展示为“d h m”格式。
3. 磁盘空间口径：展示“已用/总量 + 使用率”，单位统一为 GiB，保留 1 位小数。
4. 不可用降级：`available=false` 时展示 `--`，并在悬浮说明中显示“指标暂不可用”。

### 9.7 移动端列表显示规范（7C）

1. 断点细分：
   - 桌面：>=1080。
   - 平板窄窗：768-1079。
   - 手机：<768。
2. 列优先级：
   - P0：任务名、平台、主状态、操作入口。
   - P1：最近开播时间、录制时长。
   - P2：其余低频信息。
3. 展示策略：
   - 手机仅展示 P0，P1/P2 放入行展开区。
   - 平板默认展示 P0+P1，P2 放入行展开区。
4. 触控规范：可点击区域不小于 44x44 px，行内主操作优先保证单手可达。
5. 滚动策略：优先行展开，不以横向滚动作为默认主路径。
